构建高性能全栈应用：Turborepo + tRPC + Next.js 的 Monorepo 实践

📅 2024.10.20 ⏱️ 9 min 👤 Eric Pan

为什么要用 Monorepo

在构建复杂的全栈应用时，前后端代码通常分散在不同的仓库中，导致类型定义不同步、部署流程割裂、代码复用困难等问题。Monorepo 架构将所有相关代码放在同一个仓库中，通过 Turborepo 的智能缓存和并行构建解决多包管理的性能问题，再配合 tRPC 实现端到端的类型安全 API 调用。

本文将分享我们在一个 SaaS 产品中落地 Turborepo + tRPC + Next.js 的完整实践，包括项目结构设计、包间依赖管理、CI/CD 优化等关键环节。

Monorepo 不是银弹，但对于需要频繁跨前后端修改的团队来说，它带来的开发体验提升是显著的。

Monorepo 项目结构

我们采用 Turborepo 作为 Monorepo 的编排引擎，项目结构如下：

  • apps/web — Next.js 14 前端应用，使用 App Router
  • apps/api — tRPC 服务端，集成 Prisma ORM
  • apps/admin — 管理后台，共享 web 的组件和 tRPC 客户端
  • packages/ui — 共享 UI 组件库（基于 Radix UI + Tailwind）
  • packages/db — Prisma schema 和数据库客户端
  • packages/shared — 共享工具函数和类型定义

Turborepo 的核心优势在于其增量构建和远程缓存能力。在我们的项目中，turbo build 的缓存命中率达到了 87%，CI 构建时间从 12 分钟缩短到 3 分钟。

tRPC 类型安全 API

tRPC 最大的魅力在于它消除了 API 层的类型鸿沟。前端调用后端 API 时，IDE 会自动补全路由名称、输入参数和返回类型——不需要任何代码生成步骤。

我们采用了分层 Router 设计，将业务逻辑按领域拆分：

  • userRouter — 用户注册、登录、Profile 管理
  • projectRouter — 项目的 CRUD、权限管理
  • billingRouter — 订阅管理、账单查询、Stripe 集成
  • analyticsRouter — 数据统计、报表生成

通过 mergeRouters 将它们合并为一个统一的 AppRouter，前端只需要一个 hooks 集合就能调用所有 API。

tRPC Router 定义

以下是项目 Router 的核心实现，展示了输入校验、中间件鉴权和数据库操作的完整流程：

server/routers/project.ts
import { z } from 'zod';
import { router, protectedProcedure } from '../trpc';
export const projectRouter = router({
list: protectedProcedure
.input(z.object({
page: z.number().min(1).default(1),
limit: z.number().min(1).max(100).default(20),
}))
.query(async ({ ctx, input }) => {
const projects = await ctx.db.project
.findMany({
where: { ownerId: ctx.user.id },
take: input.limit,
skip: (input.page - 1) * input.limit,
orderBy: { createdAt: 'desc' },
});
return { projects, page: input.page };
}),
create: protectedProcedure
.input(z.object({
name: z.string().min(1).max(100),
description: z.string().optional(),
}))
.mutation(async ({ ctx, input }) => {
return ctx.db.project.create({
data: { ...input, ownerId: ctx.user.id },
});
}),
});

Turborepo 缓存策略

Turborepo 的缓存机制是其核心竞争力。它通过对每个 task 的输入（源文件 + 依赖 + 环境变量）计算哈希值，决定是否可以复用上次的构建产物。

  • 本地缓存 — 默认启用，构建产物存储在 node_modules/.cache/turbo 中，开发者本地构建提速 3-5 倍
  • 远程缓存 — 通过 Vercel Remote Cache 或自建 Turborepo Remote Cache Server 实现团队共享缓存
  • Task 依赖拓扑 — 通过 turbo.json 的 dependsOn 配置，确保上游包先于下游包构建
  • 环境变量过滤 — 使用 globalEnv 和 env 字段精确控制缓存失效条件，避免不必要的重建

配置示例中，我们将 build task 声明为依赖上游的 ^build，确保 packages/ui 先编译完成后再构建 apps/web。同时，dev task 设置 cache: false，避免开发模式的热更新被缓存拦截。

部署策略

Monorepo 的部署需要精心设计，避免一次 commit 触发所有应用的重新部署。我们采用了以下策略：

  • 变更检测 — 利用 Turbo 的 --filter 语法，只构建和部署受影响的应用
  • 独立部署 — web 和 admin 部署到 Vercel，api 部署到 AWS ECS，各自独立发布
  • 数据库迁移 — Prisma migrate 作为独立的 CI 步骤，在 API 部署前自动执行
  • 灰度发布 — 通过 Vercel Preview 和 ECS Blue/Green 实现渐进式发布
实践中最大的教训是：Monorepo 的 CI/CD 设计需要与代码结构同步演进，初期可以简单粗暴，但随着团队规模增长，必须投入时间优化构建流水线。

性能对比数据

以下是我们在 50 人开发团队中实测的数据对比：

  • 全量构建时间 — 从 12 分钟（多仓库）降低到 3 分钟（Turborepo 缓存命中）
  • CI 运行时间 — 从平均 8 分钟降低到 2.5 分钟
  • API 类型错误 — 前后端类型不一致的 bug 从每月 15+ 降低到接近零
  • 新功能开发效率 — 涉及前后端改动的需求，开发周期平均缩短 30%

tRPC 的端到端类型安全不仅减少了 bug，更重要的是提升了开发者的信心。重构 API 时，编译器会精确告诉你哪些前端代码需要更新，而不是等到运行时才发现问题。

总结

Turborepo + tRPC + Next.js 的组合为全栈 TypeScript 开发提供了一个高度集成且开发体验优秀的方案。关键优势在于：

  • 统一的代码仓库简化了跨前后端的协作和代码复用
  • 智能缓存大幅缩短了构建和 CI 时间
  • 端到端类型安全消除了 API 层的大量潜在错误
  • 模块化的包结构使得团队可以并行开发互不干扰

当然，这套方案也有其局限性——它要求整个技术栈统一为 TypeScript，对于使用多语言后端的团队可能不太适合。但对于全栈 JS/TS 团队来说，这可能是目前开发体验最好的架构选择。
