系统架构

别再把业务系统做成 CRUD 套壳

📅 2026.04 ⏱️ 9 min 👤 Eric Pan

CRUD 没错，但它不该成为系统中心

CRUD 是企业系统里绕不开的基础能力。用户、角色、字典、基础档案、分类标签这些模块，用增删改查解决很合理。问题不在 CRUD 本身，而在很多复杂业务系统最后只剩下 CRUD：建表、写接口、做列表、加表单、塞按钮，然后把真正的业务流程丢给用户自己理解。

当系统还只是数据维护工具时，这种方式足够快。但一旦进入审批、工单、项目协作、风险处理、周报生成、客户跟进这类场景，用户真正要的就不是“改一条记录”，而是完成一个任务。系统如果只提供表格和表单，就会把流程判断、风险识别和下一步行动都外包给人。

AI Agent 让这个问题更明显。Agent 要调用的是清晰的业务能力，而不是模糊的底层数据操作。一个只暴露 updateTask、saveReport、listProject 的系统，很难让 Agent 安全地完成“分析项目延期风险并生成周报”这种任务。

从数据表中心转向业务能力中心

传统 CRUD 系统往往从表结构出发：先有 project、task、report，再生成接口和页面。业务复杂度上来后，Service 层开始堆满状态判断、权限判断、通知逻辑和异常处理。代码看起来分层清楚，实际业务语义却被淹没在 updateStatus 和 saveSomething 里。

更好的方式是把接口设计成业务动作。订单不是只需要 updateOrder，而是需要 pay、cancel、ship、applyRefund；任务不是只需要 updateTask，而是需要 assign、start、submit、reviewPass、markRisk。这样的动作能表达业务意图，也能定义权限、前置条件、副作用和审计记录。

对 AI Agent 来说，这一点尤其关键。Agent 工具应该是业务能力，而不是数据库遥控器。让 Agent 调 submitReportForApproval(reportId)，比让它调 updateReportStatus(reportId, status) 安全得多，也更容易解释和追踪。

前端也要从页面中心变成任务中心

很多后台系统的前端长得很像：菜单、列表、搜索区、表单弹窗、操作按钮。它们适合维护数据，却不一定适合推进工作。用户真正关心的是：今天有什么待办？哪个项目有延期风险？哪些审批快超时？系统建议我下一步做什么？

所以复杂业务前端不应该只是“页面集合”，而应该有任务工作台。一个项目协作系统的核心入口，不该只是项目列表和任务列表，而应该呈现项目进度、本周完成、延期任务、阻塞风险、待审批事项、最近事件和 Agent 建议。

这不是 UI 包装问题，而是系统职责变化。前端从“让用户找到表并操作数据”，变成“帮助用户理解当前局面并完成任务”。当 Agent 进入系统后，前端还要承担人机确认：哪些建议只是草稿，哪些动作会改变业务状态，哪些高风险操作必须确认。

状态机、工作流和事件才撑得住复杂业务

很多系统用一个 status 字段管理复杂状态：草稿、待审批、已通过、已驳回、已归档。字段本身没问题，但如果没有合法状态迁移，任何状态都可能被接口改出来。真正需要建模的是：谁能在什么条件下，把对象从哪个状态推进到哪个状态。

状态机解决的是“状态如何变化”，工作流解决的是“流程如何推进”，事件驱动解决的是“发生了什么以及谁要响应”。三者叠起来，复杂业务才不会全部挤进一个巨大的 Service 方法。

比如任务延期不该只是把 status 改成 delayed。更完整的设计是发布 TaskDelayedEvent，由风险服务更新项目风险，由通知服务提醒项目经理，由周报服务沉淀素材，由 Agent 服务生成延期原因摘要。这样系统记录的不只是最终结果，还有业务过程。

RAG 和 Agent 需要知识层与工具边界

智能业务系统不能只存最终数据，还要沉淀过程知识：需求文档、会议纪要、任务评论、审批意见、历史周报、风险记录、项目复盘。这些内容进入知识库后，RAG 才能让 Agent 基于真实上下文生成总结、建议或报告。

但 RAG 不是把文档扔进向量库就结束。业务系统还要处理权限、版本、来源标注、更新时间和引用展示。不同项目、部门、角色能访问的文档不同，Agent 检索时也必须遵守同样的权限边界。

工具层同样需要边界。低风险工具可以生成周报草稿、分析风险、总结会议；高风险工具，比如提交审批、发送邮件、关闭项目、修改负责人，必须要求人类确认。Agent 可以建议行动，但不应该绕过业务权限和审计机制直接改系统。

写在最后

跳出 CRUD 不是抛弃 CRUD，而是别让 CRUD 成为复杂系统的唯一抽象。CRUD 负责基础数据维护，业务能力层负责表达动作，状态机和工作流负责控制过程，事件驱动负责解耦协作，知识库和 Agent 负责智能辅助。

未来有竞争力的前后端系统，不会只是“表格 + 表单 + 接口”的组合，而会更像一个业务能力平台：前端是任务工作台，后端暴露可授权、可审计、可编排的业务动作，数据层沉淀过程和知识，Agent 在可控边界内参与任务推进。

真正的系统升级，不是给 CRUD 页面外挂一个聊天框，而是把系统从“管理数据”升级为“推进业务”。AI Agent 时代，最值钱的不是让模型会说话，而是让它能在清晰边界里调用业务能力，帮助用户把复杂工作稳定做完。
