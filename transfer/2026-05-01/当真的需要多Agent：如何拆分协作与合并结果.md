# 当真的需要多 Agent：如何拆分、协作与合并结果

多 Agent 不是不能用。真正的问题是，很多系统还没有搞清楚为什么要拆，就已经开始设计一堆角色。

一个复杂任务来了，先写出“规划 Agent、执行 Agent、审核 Agent、反思 Agent、总结 Agent”，看起来像一个完整团队，但实际上只是把一个问题拆成了多个会说话的节点。系统表面变复杂了，推理并没有更稳定；流程变长了，结果并没有更可靠。

多 Agent 的价值，不在于“多”，而在于它能不能把复杂任务拆成可控制、可追踪、可组合的协作单元。

所以，真正需要讨论的不是“要不要多 Agent”，而是更具体的问题：

> 什么时候必须拆？  
> 按什么边界拆？  
> Agent 之间共享什么？  
> 结果如何合并？  
> 谁负责最终判断？

如果这些问题没有回答清楚，多 Agent 只会把一个模型的不确定性，扩散成一组模型的不确定性。

## 一、不是不能拆，而是不能无边界地拆

单 Agent 有明显优势：上下文集中，目标一致，推理主线连续。很多任务只要单 Agent 能稳定完成，就不应该拆。强行拆分，会引入额外通信成本、状态同步成本和结果合并成本。

但单 Agent 也有边界。

当任务很长、上下文很重、工具很多、风险等级不同、需要多角度验证，或者存在天然并行子任务时，单 Agent 会变得吃力。它既要规划，又要检索；既要生成，又要审查；既要执行，又要判断风险。所有能力混在一个上下文里，容易造成注意力分散、上下文污染和执行边界不清。

这时，多 Agent 才有意义。

但多 Agent 不是为了模仿人类组织架构，而是为了把任务内部的复杂度重新组织。拆分的目的不是增加角色，而是降低单点上下文负担、隔离风险、引入并行、增强验证，并让长期任务的状态更清楚。

换句话说：

> 多 Agent 不是把任务拆碎，而是把任务拆成可以协作的结构。

## 二、什么时候真的需要多 Agent

判断是否需要多 Agent，不能只看任务“复杂不复杂”。复杂任务也可能适合单 Agent，简单任务也可能因为权限或工具边界而需要拆。

真正值得拆的场景，通常有五类。

### 1. 任务天然可以并行

如果多个子任务彼此独立，不需要实时共享中间状态，就适合拆。

比如同时总结多篇文章、分析多个文件、检索多个资料源、评估多个候选方案。这类任务中，每个子 Agent 都可以在局部上下文中完成工作，最后返回结构化结果。

这种拆分的价值是并行和压缩。主 Agent 不必把所有细节都塞进同一个上下文，而是只接收每个子任务的结论、证据和必要摘要。

### 2. 子任务需要不同工具或能力

一个完整任务可能同时包含文档检索、代码分析、数据计算、网页查询、文件修改、仓库提交、知识库写入。不同步骤需要的工具不同，风险等级也不同。

如果所有工具都交给一个 Agent，自由度会过高，边界会模糊。更合理的方式，是把工具能力按边界拆开：检索单元只负责找依据，分析单元只负责判断，执行单元只负责受控修改，审核单元只负责检查风险。

这里的拆分不是角色扮演，而是工具边界设计。

### 3. 需要独立视角进行验证

有些任务需要第二视角，不是为了并行，而是为了校验。

比如方案生成之后，需要风险审查；代码修改之后，需要代码审查；研究文章生成之后，需要事实核查；系统设计之后，需要反向质疑。

这类多 Agent 的核心价值，不是让多个 Agent 同时工作，而是避免同一个 Agent 在自己的假设里自我强化。

生成者容易维护自己的逻辑，审查者必须站在约束、风险和证据一侧。二者的目标不同，才有拆分价值。

### 4. 需要隔离权限和风险

多 Agent 最现实的价值之一，是权限隔离。

读文件、查资料、运行脚本、修改代码、删除数据、发送邮件、提交仓库，这些动作风险完全不同。如果一个 Agent 同时拥有全部能力，那么一次错误判断就可能直接产生高风险结果。

更稳妥的架构是：低风险 Agent 可以自由分析，高风险 Agent 必须受控执行；只读 Agent 不能写入；写入 Agent 必须拿到明确指令；发布类 Agent 必须经过人工确认或审核机制。

这里的拆分，不是为了“智能”，而是为了安全。

### 5. 任务生命周期很长，需要状态分工

长期任务不是一次问答，而是持续推进。它会产生计划、执行记录、失败原因、阶段结论、用户确认、知识沉淀。

如果所有状态都堆在一个 Agent 对话里，很快会混乱。多 Agent 可以让不同单元围绕统一状态工作：有的负责规划，有的负责执行，有的负责监控，有的负责复盘，有的负责写回知识库。

但前提是必须有统一状态源。否则长期协作会变成多份上下文各自分叉。

## 三、Agent 应该按什么边界拆

多 Agent 最差的拆法，是按人类岗位机械拆分。比如“产品经理 Agent、架构师 Agent、程序员 Agent、测试 Agent”。这类拆法表面清晰，实际往往没有定义信息流、状态流和结果流。

更合理的拆分，应当围绕工程边界。

### 1. 按任务依赖拆

如果两个子任务可以独立完成，就可以拆。如果一个子任务的判断会持续改变另一个子任务的前提，就不能简单隔离。

能并行的任务适合 Worker 式拆分；强依赖任务要么保持单 Agent，要么使用共享状态的 Team 机制。

判断标准很简单：

> 子任务之间如果只需要交换结果，可以拆。  
> 子任务之间如果必须共享过程，就不能只靠分发和回收。

### 2. 按上下文边界拆

每个 Agent 不应该拿到全部上下文，而应该拿到完成任务所需的最小充分上下文。

上下文太少，会误判。上下文太多，会污染。真正的设计能力，在于判断哪些信息必须原文保留，哪些信息可以摘要传递，哪些信息根本不该传。

例如，事实核查 Agent 需要原始出处和待核查结论，不一定需要完整写作过程；写作 Agent 需要论点、结构和关键素材，不一定需要全部检索噪音；执行 Agent 需要明确指令和目标文件，不应该获得无关历史对话。

上下文不是越多越好。上下文要有边界。

### 3. 按工具边界拆

工具是 Agent 作用于外部世界的接口。工具越有副作用，越需要独立边界。

检索工具、文件读取工具、代码执行工具、数据库写入工具、GitHub 提交工具、邮件发送工具，不应被视为同一类能力。

工具边界的原则是：

> 低风险工具可以开放，高风险工具必须收敛；只读能力可以前置，写入能力必须受控。

这会让 Agent 系统从“能做事”走向“可控地做事”。

### 4. 按状态所有权拆

多 Agent 系统必须回答一个问题：谁维护任务的真实状态？

如果每个 Agent 都维护一份自己的状态，系统必然分叉。一个 Agent 认为任务已经完成，另一个 Agent 认为还在分析；一个 Agent 使用旧约束，另一个 Agent 使用新约束；一个 Agent 已经发现风险，另一个 Agent 继续按原计划执行。

所以多 Agent 系统需要状态所有权：可以是 Coordinator，可以是共享状态板，也可以是结构化任务记录。但必须有一个地方保存当前目标、约束、决策、进度和未解决问题。

没有状态所有权，多 Agent 只是多个上下文在同时漂移。

### 5. 按结果对象拆

有些任务可以按输出对象拆。比如一份报告的不同章节、一个系统的不同模块、一个项目的不同风险项、一个知识库的不同文档类型。

这种拆法适合输出对象相对独立的场景。但最终仍然需要统一标准、统一风格、统一主线。否则每个部分都能看，整体却不成立。

## 四、Agent 之间应该共享什么

多 Agent 协作不是完全共享，也不是完全隔离。共享过少，协作断裂；共享过多，噪音污染。

关键是共享必要信息。

必须共享的内容包括：任务目标、全局约束、关键事实、当前状态、已确认决策、输出标准、风险边界。这些信息决定多个 Agent 是否在做同一件事。

可以压缩共享的内容包括：历史对话、资料摘要、中间分析、候选方案、失败尝试、长文档内容。这些信息不一定完整传递，但需要以摘要、引用或结构化字段传递。

不应该随意共享的内容包括：无关上下文、未验证结论、低质量推测、过多草稿、高风险工具权限、与当前子任务无关的历史细节。

一个好的原则是：

> 每个 Agent 只获得完成当前任务所需的最小充分上下文。

这句话看起来简单，但它决定了多 Agent 系统是否干净。很多多 Agent 系统的问题，不是信息不够，而是信息太乱。所有 Agent 都看到一堆不该看的内容，最后每个 Agent 都在不同噪音里做判断。

## 五、几种常见的协作结构

多 Agent 不只有一种形态。不同任务结构，应该对应不同协作拓扑。

### 1. Coordinator-Worker：主控加子任务

这是最常见的结构。Coordinator 负责目标、计划、上下文分发和结果合并，Worker 负责局部任务。

它适合并行检索、多文件分析、多方案生成、批量检查。优点是结构清楚，子任务隔离。缺点是 Coordinator 的合并压力很大，Worker 返回的信息如果过度压缩，容易丢失关键依据。

### 2. Pipeline：流水线处理

Pipeline 是一个 Agent 的输出成为下一个 Agent 的输入。

比如：检索 → 分析 → 生成 → 审查 → 写入。它适合文档处理、代码生成、数据分析、知识反写等流程化任务。

它的风险是错误传递。前面一步错了，后面每一步都可能建立在错误前提上。所以 Pipeline 必须有校验点、回退机制和中间状态记录。

### 3. Blackboard：共享状态板

Blackboard 模式中，多个 Agent 不一定直接互相对话，而是围绕一个共享状态区工作。它们读取任务状态，写入发现、风险、候选方案和决策。

这种结构适合复杂系统设计、多轮诊断、长期协作任务。它的优势是状态集中，缺点是状态板容易污染，因此需要写入权限、版本管理和冲突处理。

### 4. Generator-Critic：生成与审查

一个 Agent 负责生成方案，另一个 Agent 负责审查风险、事实、逻辑或规范。它适合代码审查、文章修订、方案评估、事实核查。

这种结构的关键是审查标准必须明确。否则 Critic Agent 很容易变成泛泛挑错，提出看似合理但没有约束价值的意见。

## 六、多个 Agent 的结果如何合并

多 Agent 系统的结果合并，不能只是把输出拼接起来。拼接不是合并，堆叠不是综合。

不同任务需要不同合并方式。

### 1. 汇总式合并

适合独立子任务，比如多个资料源摘要、多个文件分析、多个模块检查。合并时要统一格式、保留来源、去重、标注冲突。

这种合并的重点是完整和清晰。

### 2. 裁决式合并

适合多个 Agent 给出不同判断的场景。比如不同方案评估、不同风险判断、不同事实结论。

裁决不能靠“最后一个 Agent 感觉哪个更好”。必须有标准：证据优先、约束优先、用户目标优先、风险优先，或者明确的评分规则。

裁决 Agent 可以存在，但它不能只是总结员，而应该是基于规则和证据的判断者。

### 3. 综合式合并

适合文章、报告、系统方案这类强主线输出。合并不是把多个段落粘起来，而是提取共识、识别冲突、保留证据、重建结构、统一表达。

在这种场景下，最终合并者必须维护主线。否则多 Agent 的产物会变成局部正确、整体松散。

### 4. 状态更新式合并

适合长期任务。多个 Agent 的输出不一定直接给用户，而是更新任务状态。

比如新增事实、更新计划、记录风险、标记完成项、保存失败原因、写入知识库。这类合并必须结构化，否则后续 Agent 无法复用。

多 Agent 结果合并的本质是：

> 把多个局部判断转化为一个一致、可追踪、可继续使用的状态。

这比生成一个好看的总结更重要。

## 七、必须设计可观测性和裁决机制

多 Agent 系统如果不可观测，就无法工程化。

你必须知道任务是怎么拆的，每个 Agent 收到了什么上下文，调用了哪些工具，产生了哪些中间结果，哪些结论被采用，哪些结论被丢弃，冲突如何解决，最终答案由谁合成。

否则，一旦结果出错，系统无法复盘。你只知道“多 Agent 给了一个答案”，却不知道错误来自任务拆分、上下文传递、工具调用、审查失效，还是最终合并。

多 Agent 还必须有裁决机制。

复杂任务里，冲突不是异常，而是常态。不同 Agent 给出不同结论，很正常。关键是系统必须知道如何处理分歧：保留冲突、请求更多证据、交给裁决 Agent、回退到人工确认，还是根据规则自动选择。

没有裁决机制的多 Agent，只是在等待最后一个模型把冲突糊成一段流畅的话。

## 八、结语：多 Agent 的价值在结构化协作

真正需要多 Agent 的时候，拆分不是问题；无边界拆分才是问题。

好的多 Agent 架构，不是让多个 Agent 同时发言，而是让多个上下文处理单元围绕同一个任务状态，有边界地协作。

什么时候拆，要看任务是否具备并行性、专业化边界、验证需求、权限边界或长期状态分工。  
怎么拆，要看任务依赖、上下文边界、工具边界、状态所有权和结果对象。  
共享什么，要遵循最小充分上下文。  
怎么合并，要根据任务类型选择汇总、裁决、综合或状态更新。  
能不能工程化，要看系统是否可观测、可追踪、可裁决、可恢复。

多 Agent 的成熟标志，不是 Agent 数量越来越多，而是系统边界越来越清楚。

如果单 Agent 能完成，就不要拆。  
如果子任务独立，就用 Sub-Agent。  
如果必须共享状态，就设计 Agent Team。  
如果涉及高风险操作，就做权限隔离。  
如果多个结果会冲突，就设计裁决机制。  
如果任务需要长期推进，就维护统一状态源。

最终，多 Agent 架构的核心不是“多几个智能体”，而是把复杂任务组织成一套可靠的信息流、状态流和结果流。

这才是多 Agent 从演示走向工程系统的关键。
