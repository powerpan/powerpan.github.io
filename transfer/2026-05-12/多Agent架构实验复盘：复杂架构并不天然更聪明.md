# 我做了一次多 Agent 架构实验：复杂架构并不天然更聪明

之前我写过几篇关于 AI Agent 架构的文章，核心观点大致可以概括为一句话：多 Agent 的真正问题，不是给模型分配多少角色，而是如何定义上下文边界、任务边界、协作方式和结果合并机制。

这一次，我没有继续停留在理论层面，而是做了一个小型实验平台：Agent Architecture Lab。它的目标不是解决某个具体业务问题，而是把不同 Agent 架构放在同一批任务下面，观察它们在输出质量、成本、延迟和稳定性上的差异。

实验规模不大，但结果足够有代表性。它让我更加确信一个判断：复杂架构不是质量放大器，而是成本、上下文和失败面的放大器。只有当任务本身确实需要拆解、审查或多立场权衡时，多 Agent 架构才可能值得。

## 实验怎么做

这次实验用了四类任务：

第一类是从零生成一个简单实验平台方案。
第二类是基于上一版方案扩展详细设计。
第三类是对详细设计做审查报告。
第四类是从审查报告中选择一个最有取舍的问题，并给出技术判断。

对应地，我比较了四种架构：

`single`：一个 Agent 直接完成任务。

`planner_executor`：Planner 先拆解任务，Executor 再执行。

`planner_executor_reviewer`：Planner 拆解任务，Executor 写草稿，Reviewer 审查，Executor 再生成最终答案。

`debate`：两个 Debater 分别给出方案或立场，最后由 Judge 综合。

观察指标包括耗时、token 消耗、估算成本、是否截断、最终答案可用性，以及中间过程是否有复盘价值。

这不是一个统计意义上足够严格的大样本实验。每个任务每种架构只跑了一次，所以它不适合得出“统计显著”的结论。但作为第一轮工程观察，它很有价值：它暴露了不同 Agent 架构在真实运行时会遇到的典型问题。

## Single Agent 仍然是强基线

很多人谈多 Agent 时，会默认认为 single agent 是一种低级形态，似乎只要引入 Planner、Reviewer、Debater，系统就会自然变得更强。

实验结果并不支持这种直觉。

在这次实验中，`single` 是成本和延迟最低的架构，同时稳定性也最好。四个任务里，它没有出现截断。更重要的是，在简单方案任务和明确技术取舍任务中，它的输出质量并不弱。对于边界清楚、输入充分、输出结构明确的任务，一个单 Agent 已经可以完成得很好。

这说明一个很重要的工程事实：Single Agent 不是过时方案，而是所有复杂架构必须证明自己值得超过的基线。

如果一个任务本身并不复杂，或者复杂性主要来自表达而不是推理，那么增加多个 Agent 很可能只是增加了过程成本。系统看起来更“高级”，但实际结果未必更好。

## Planner 有用，但会制造输出膨胀

`planner_executor` 的表现比较微妙。

在审查类任务中，它的效果较好。因为审查任务本身就需要覆盖多个维度，例如公平性、状态污染、并发限流、成本估算、失败记录、API key 安全、Judge 设计等。Planner 先把检查项拆出来，Executor 再逐项执行，这确实有助于提高覆盖度。

但在详细设计任务中，Planner 也暴露了明显副作用：它会推动 Executor 覆盖更多章节，输出更长、更完整，也更容易撞到输出上限。本轮实验中，`planner_executor` 在详细设计任务里出现了 `finish_reason=length`，也就是最终答案被截断。

这点很关键。Planner 的作用不应该只是“让模型多想一点”，而应该是让任务拆解更可控。如果没有章节预算、长度约束和输出边界，Planner 很容易把任务推向输出膨胀：标题更多、段落更多、结构更复杂，但最终文档反而不完整。

所以，Planner 架构要成立，必须配合预算控制。它不只是任务拆分器，也应该是输出边界管理器。

## Reviewer 的价值必须被看见

`planner_executor_reviewer` 是这次实验里成本最高、延迟最高的架构。理论上，它应该最适合需要“草稿—审查—修订”链路的任务。但本轮实验中，它没有体现出与成本匹配的稳定质量跃迁。

尤其是在详细设计任务里，即使引入 Reviewer，最终答案仍然发生截断。这说明 Review 本身并不能自动解决长文档生成的问题。相反，它还会增加更多中间文本，让后续上下文变长，进一步放大 token 消耗和输出压力。

Reviewer 并不是没有价值。它的价值在于：系统可以记录草稿是什么、审查意见是什么、最终答案改了什么。但如果这些信息没有被结构化呈现，用户只能看到最终答案，那么 Reviewer 的价值就不明显。

一个真正有工程意义的 Reviewer 架构，至少应该回答几个问题：

Reviewer 发现了哪些问题？
Executor 采纳了哪些意见？
哪些意见没有采纳，为什么？
草稿到最终稿具体发生了什么变化？

如果这些问题无法被观察，Reviewer 就只是一次昂贵的额外模型调用。

这也呼应了我之前关于多 Agent 的判断：多 Agent 架构的关键不是“多几个角色”，而是让协作过程变成可验证、可追踪、可复盘的系统结构。

## Debate 适合真实取舍，不适合所有任务

`debate` 架构在这次实验中有一个很清晰的适用场景：技术取舍判断。

在第四个任务中，系统需要从审查报告里选择一个最模糊、最有取舍的问题，并给出 A/B 理由、风险、边界、推荐决策和验证方式。这个任务天然适合 Debate，因为它本身就包含多个立场。

比如实验中出现的核心问题是 Snapshot Mode vs Chain Mode。一个方向强调公平、隔离和可复现；另一个方向强调连续性、状态传递和真实工作流。让两个 Agent 分别站在不同立场，再由 Judge 综合判断，这种结构是合理的。

但 Debate 并不适合所有任务。在简单方案任务中，Debate 容易变成“一个 Agent 给出建设性方案，另一个 Agent 补充风险，Judge 再做汇总”。这看起来像辩论，实际上更像方案扩写。它增加了成本，但未必带来真正的对抗性思考。

因此，Debate 架构的前提不是“让两个 Agent 都说一遍”，而是任务中确实存在互斥立场、权衡空间和不确定性。

没有真实分歧的任务，不需要辩论。

## 复杂架构的代价比想象中更具体

这次实验还有一个很直接的发现：复杂架构的代价不是抽象的。

`planner_executor` 的成本大约是 `single` 的 2.42 倍。
`debate` 的成本大约是 `single` 的 3.90 倍。
`planner_executor_reviewer` 的成本大约是 `single` 的 6.17 倍。

这还只是模型调用成本。实际工程中，还要考虑更多因素：结果存储、中间过程展示、失败重试、超时处理、前端可视化、人工复盘成本，以及更复杂的调试路径。

复杂架构一旦引入，就不只是多几个 prompt，而是引入一套新的系统复杂度。

这也是为什么我认为，多 Agent 架构不能被当成默认升级路线。它应该是按任务类型选择出来的结构，而不是为了“看起来更智能”而堆出来的结构。

## 一套更实际的选择方式

从这次实验看，不同任务适合的架构可以粗略归纳为：

简单方案、边界清晰、输出要求明确的任务，优先使用 `single`。它便宜、快、稳定，而且质量未必差。

中等复杂度、多维度覆盖的任务，可以考虑 `planner_executor`。尤其是审查、评估、清单式分析这类任务，Planner 有助于提升覆盖度。

需要保留草稿、审查、修订证据链的任务，才适合使用 `planner_executor_reviewer`。但前提是系统必须展示 Review 如何改变最终结果，否则它的价值很难被感知。

存在真实分歧、A/B 取舍、路线判断的问题，适合 `debate`。它不是为了制造热闹，而是为了组织不同立场之间的冲突和综合。

换句话说，问题不应该是“我要不要使用多 Agent”，而应该是：

这个任务是否真的需要拆解？
是否真的存在多个立场？
是否需要审查证据链？
中间过程是否会被使用？
增加的 token、延迟和失败面是否值得？

只有这些问题成立，多 Agent 才有工程意义。

## 回到架构本质

这次实验让我更明确地看到，多 Agent 架构的价值不是让模型突然变强，而是把复杂任务里的不确定性组织起来。

Planner 组织任务结构。
Reviewer 组织质量反馈。
Debate 组织立场冲突。
Judge 组织最终合并。

但这些结构都有成本，也都有失败方式。Planner 可能导致输出膨胀，Reviewer 可能变成昂贵的形式主义，Debate 可能退化成一主一辅的方案拼接，Judge 也可能偏向某一方而不自知。

所以，多 Agent 的成熟，不是从一个 Agent 走向越来越多 Agent，而是知道什么时候应该保持简单，什么时候才值得引入复杂协作。

如果说之前的理论文章讨论的是多 Agent 架构的边界，那么这次实验给我的实践结论是：

多 Agent 架构不是智能的堆叠，而是复杂性的管理。
真正的工程判断，不是能不能加 Agent，而是加了以后，是否真的更可控、更可靠、更值得。
