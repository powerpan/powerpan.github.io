# 多 Agent 架构的真正问题：不是角色分工，而是上下文边界

很多人设计多 Agent 系统时，第一反应是“分角色”。

一个复杂任务来了，就拆成规划 Agent、检索 Agent、写作 Agent、审核 Agent、执行 Agent、总结 Agent。看起来像一个完整团队，每个 Agent 都有明确身份，系统似乎也因此变得更“智能”。

但真实情况往往相反：Agent 越多，系统越乱；角色越细，结果越碎；流程看似复杂，效果却不如一个单 Agent 稳定。

这就是很多多 Agent 系统“一拆就废”的原因。

问题通常不在于模型能力不足，而在于架构把原本需要连续理解的上下文切碎了。任务被拆开之后，每个 Agent 只拿到局部信息，各自做出看似合理的判断，但最终合并起来却失去主线、互相冲突、重复分析，甚至丢掉关键约束。

所以，多 Agent 架构的核心问题，不是“应该设计几个角色”，而是“任务之间的上下文边界应该怎么划”。

多 Agent 的设计单位不是角色，而是上下文。

## 一、错误的起点：把多 Agent 当成角色扮演

很多多 Agent 设计，本质上是在模仿人类团队。

比如：

```text
产品经理 Agent
架构师 Agent
程序员 Agent
测试 Agent
审稿 Agent
反思 Agent
```

这种设计看起来很自然。因为在现实组织里，复杂任务确实需要不同角色协作。产品经理理解需求，架构师设计方案，工程师实现功能，测试人员保证质量。

但问题在于，Agent 不是人类团队的简单复制。

人类团队协作时，有大量隐性上下文：长期经验、项目背景、组织共识、责任意识、非语言信息、历史决策。一个工程师即使只收到一句简单需求，也能基于过去的上下文补全很多信息。

Agent 不一样。Agent 的协作高度依赖显式上下文。如果上下文没有被正确传递，它就不会“默认知道”。如果状态没有被同步，它就会基于过期信息判断。如果边界没有设计好，它就会重复、跑偏或者互相干扰。

因此，把多 Agent 理解成“多个虚拟员工分工”，很容易走错方向。

更准确的理解应该是：

> 多 Agent 不是把一个任务分给多个角色，而是把一个推理过程拆成多个上下文处理单元。

角色只是表象。真正重要的是信息如何流动、状态如何共享、结论如何合并。

## 二、为什么很多多 Agent 系统一拆就废

复杂任务并不天然适合拆分。

有些任务看起来包含很多步骤，但它们依赖同一条连续主线。比如写一篇观点文章、设计一个系统架构、分析一个复杂 Bug、规划一个产品方案。这类任务需要持续维护目标、约束、判断标准和上下文一致性。

如果过早拆成多个 Agent，就会出现三个问题。

第一，主线被切断。

每个子 Agent 都只看到任务的一部分。它可能完成了局部任务，但不一定理解整体意图。最后得到的结果就像拼接出来的碎片：每段都有道理，但整体没有方向。

第二，信息在传递中损耗。

主 Agent 给子 Agent 分配任务时，通常会压缩上下文。子 Agent 返回结果时，又会再压缩一次。每一次摘要都会丢失信息。问题在于，被丢掉的往往不是无关信息，而可能是后续判断需要的关键约束。

第三，协作产生噪音。

多个 Agent 如果没有清晰边界，就会重复分析同一个问题，给出不同版本的解释，互相纠偏，制造大量中间内容。表面上看系统更“会思考”，实际上只是把任务变得更吵。

这就是多 Agent 的悖论：它本来是为了解决复杂任务，却可能制造新的复杂度。

Agent 拆分不是免费增强。拆分会减轻单个 Agent 的上下文压力，但会增加通信、同步、合并和一致性成本。

换句话说：

> Agent 拆分不是消灭复杂度，而是把复杂度从单个上下文内部，转移到多个上下文之间。

如果没有意识到这一点，多 Agent 很容易从“协作系统”变成“噪音系统”。

## 三、真正关键：任务之间是否需要共享上下文

判断一个任务是否适合多 Agent，最重要的问题不是“任务复杂不复杂”，而是：

> 子任务之间是否需要共享上下文？

如果任务之间几乎不互相影响，那么拆分是合理的。  
如果任务之间必须持续共享状态，那么简单拆分就是危险的。  
如果单个 Agent 能稳定完成任务，那就没有必要为了架构复杂度而拆分。

可以把任务分成两类。

### 低上下文依赖任务

这类任务适合拆。

它们的特点是：

```text
子任务相互独立
不需要实时共享状态
每个子任务只需要局部信息
最终只需要返回结论
```

比如，让不同 Agent 分别总结几篇文章；让不同 Agent 检查多个文件；让不同 Agent 从不同资料源检索信息；让多个 Agent 对同一问题给出独立判断。

这类任务适合 Sub-Agent。

主 Agent 可以把任务分发出去，子 Agent 在隔离上下文中完成工作，只返回压缩后的结论。它们不需要理解整个任务全貌，也不需要和其他 Agent 持续互动。

这种拆分是有价值的，因为它带来了隔离、并行和上下文压缩。

### 高上下文依赖任务

这类任务不能简单拆。

它们的特点是：

```text
任务之间持续互相影响
一个判断会改变另一个判断的前提
需要共享状态
需要共同维护任务主线
需要反复修正和协同
```

比如复杂系统设计、长文写作、代码重构决策、多步骤问题诊断、涉及状态变化的执行任务。这些任务不是多个独立子任务的简单相加，而是一条不断演化的推理链。

如果这类任务被粗暴拆分，结果往往会失控。

写作 Agent 可能写得很好，但偏离整体论点；架构 Agent 可能设计得合理，但没有考虑业务约束；测试 Agent 可能指出问题，但不知道前面为什么做了某种取舍。

这类任务如果确实需要多 Agent，就不能采用简单的 Sub-Agent 模式，而应该更接近 Agent Team：多个 Agent 共享状态、围绕同一个任务主线持续协作，并且有明确的状态同步和冲突处理机制。

所以，多 Agent 架构的第一判断不是“要几个 Agent”，而是：

> 这些 Agent 是否必须看到同一份上下文？

## 四、单 Agent、Sub-Agent、Agent Team 的边界

多 Agent 架构可以粗略分成三种形态：单 Agent、Sub-Agent、Agent Team。

它们不是高级程度不同，而是适用的问题结构不同。

### 单 Agent：适合主线强、上下文必须连续的任务

单 Agent 的优势是上下文集中、目标一致、判断连续。

如果一个任务要求保持统一风格、统一逻辑、统一判断标准，那么单 Agent 往往比多 Agent 更稳。

比如写一篇观点文章。文章不是资料摘要的拼接，而是围绕一个中心论点展开。如果拆成“资料 Agent、结构 Agent、写作 Agent、润色 Agent”，每个 Agent 都可能完成局部工作，但文章的思想主线可能被稀释。

同样，很多系统架构判断也不适合一开始就拆。因为模块划分、数据模型、状态流转、权限边界之间互相影响。过早拆分会让每个 Agent 只优化局部，却破坏整体一致性。

单 Agent 的问题是上下文可能过长，任务负担较重。但只要它能稳定完成，就不应该为了“看起来先进”而拆分。

### Sub-Agent：适合可隔离、可并行、只需返回结论的任务

Sub-Agent 的核心是隔离。

主 Agent 负责总体目标和主线控制，子 Agent 只处理边界清晰的局部任务。子 Agent 不需要知道所有上下文，也不需要参与全局决策，只需要返回一个可用结论。

比如：

```text
分别总结十篇文章
分别检查十个模块
分别从多个资料源检索证据
分别评估几种方案的优缺点
```

这种模式的优势是并行和压缩。每个子 Agent 可以在自己的局部上下文中工作，避免主 Agent 的上下文过载。

但 Sub-Agent 的边界必须清晰。如果子任务需要频繁参考其他子任务的判断，或者子任务结果会持续改变整体方向，那么它就不适合简单隔离。

### Agent Team：适合必须共享状态、持续协作的任务

Agent Team 的核心是共享状态。

它适合那些多个 Agent 必须围绕同一份任务状态持续互动的场景。每个 Agent 的判断都可能影响其他 Agent 的输入，任务不是一次分发、一次回收，而是多轮协作、动态修正。

例如，一个复杂系统的设计过程可能需要需求分析、架构设计、风险评估、实现约束之间不断互相校正。此时，如果每个 Agent 都完全隔离，就会导致方案互相矛盾。更合理的方式是让它们共享关键状态，并在统一任务上下文中协作。

但 Agent Team 的代价也最高。

它需要处理状态同步、冲突解决、协作顺序、消息控制、上下文污染和最终裁决。没有这些机制，Agent Team 很容易变成一群 Agent 在同一个问题上反复发言。

所以，Agent Team 不是“多叫几个 Agent 来讨论”，而是一种更复杂的共享上下文系统。

## 五、多 Agent 的核心成本：信息丢失、协作噪音、状态不一致

多 Agent 架构真正困难的地方，不是让多个 Agent 同时工作，而是控制它们之间的副作用。

### 信息丢失

上下文传递不是无损的。

主 Agent 不可能把所有信息完整交给每个子 Agent。子 Agent 也不可能把所有推理过程完整返回给主 Agent。中间一定会发生压缩、摘要和筛选。

这会带来一个风险：系统不知道自己丢了什么。

在普通任务中，这可能只是结果不够细。  
在复杂任务中，这可能导致判断方向错误。

### 协作噪音

多个 Agent 并不必然带来更好的判断。

如果边界不清，它们会重复讨论同一问题，输出大量相似观点，甚至互相否定。系统看起来更复杂，但有效信息密度下降。

协作不是越多越好。协作本身也是成本。

一个好的多 Agent 系统，不应该追求“更多声音”，而应该追求“更少但更必要的信息交换”。

### 状态不一致

状态不一致是多 Agent 系统最危险的问题之一。

一个 Agent 认为目标是 A，另一个 Agent 认为目标是 B；一个 Agent 使用旧约束，另一个 Agent 使用新约束；一个 Agent 已经基于错误前提继续推理，另一个 Agent 却没有意识到。

在代码修改、任务执行、知识反写、系统诊断这类场景里，状态不一致会直接破坏可靠性。

所以，多 Agent 系统必须有清晰的状态管理机制。否则，它不是协作，而是分裂。

## 六、先设计信息流，再设计角色名

多 Agent 架构的设计顺序应该反过来。

不要一开始就问：

```text
我要几个 Agent？
每个 Agent 叫什么？
每个 Agent 扮演什么角色？
```

而应该先问：

```text
这个任务的主线是否必须连续？
哪些信息必须共享？
哪些信息可以隔离？
哪些子任务可以独立完成？
哪些结论需要回传？
谁负责最终裁决？
状态由谁维护？
冲突如何解决？
```

角色名可以最后再定。甚至很多时候，角色名并不重要。

真正重要的是信息流。

一个可靠的多 Agent 系统，应该能回答几个基本问题：

```text
信息从哪里来？
传给谁？
传多少？
以什么形式传？
哪些信息不应该传？
中间结果如何合并？
冲突由谁解决？
最终状态在哪里维护？
```

这些问题决定了系统是否稳定，而不是“有几个 Agent”决定系统是否稳定。

所以，多 Agent 设计的原则可以简化成一句话：

> 先判断上下文依赖，再决定 Agent 数量；先设计信息流，再设计角色分工。

## 七、最小拆分原则

多 Agent 系统还应该遵循一个朴素原则：能不拆就不拆。

这不是保守，而是工程理性。

单 Agent 能稳定完成，就不要拆。  
子任务能隔离完成，再用 Sub-Agent。  
任务必须共享状态，才考虑 Agent Team。  
只是为了让系统看起来复杂，不应该拆。

很多工程系统的问题，不是能力不够，而是抽象过度。多 Agent 也是一样。

把简单任务拆复杂，会引入不必要的通信成本。  
把连续任务拆断，会破坏上下文一致性。  
把强依赖任务拆成独立 Agent，会制造状态冲突。  
把单 Agent 能完成的任务硬拆成多 Agent，只是在制造架构噪音。

真正成熟的架构设计，不是能拆多细，而是知道哪里不该拆。

## 八、结语：多 Agent 的本质是上下文边界设计

多 Agent 架构真正要解决的，不是角色分工问题，而是上下文边界问题。

低水平的多 Agent 设计，先想角色：谁负责规划，谁负责写作，谁负责审核，谁负责执行。  
高水平的多 Agent 设计，先看上下文：哪些信息必须连续，哪些任务可以隔离，哪些状态必须共享，哪些结果只需要汇总。

如果任务无需共享信息，Sub-Agent 是合理的。它通过隔离、压缩和并行，提高局部任务处理效率。

如果任务必须共享状态、持续协作、实时互相影响，就需要 Agent Team。但 Agent Team 必须处理状态同步、协作噪音和冲突裁决，否则只会让系统更乱。

如果单个 Agent 能完成任务，就不需要拆。因为拆分本身不是价值，只有正确的上下文边界才是价值。

多 Agent 系统真正难的地方，不是让多个 Agent 工作，而是让它们在正确的上下文边界内工作。边界设计对了，拆分才有意义；边界设计错了，Agent 越多，系统越差。

最终，多 Agent 架构的关键判断不是：

```text
这个任务需要几个 Agent？
```

而是：

```text
这个任务的上下文应该如何被共享、隔离、压缩、传递和合并？
```

这才是多 Agent 从玩具走向工程系统的分界线。
