# 如果没有数据库，系统如何设计？

很多业务系统的设计，几乎都是从数据库开始的。

需求刚出来，第一反应往往不是问：业务流程是什么？状态如何流转？哪些事实需要被记录？哪些规则必须被约束？而是先问：

```txt
需要几张表？
字段怎么设计？
主键外键怎么连？
接口怎么写？
```

这当然不是错误。数据库是现代系统最重要的基础设施之一。它稳定、成熟、可查询、可事务化，能快速支撑大量业务开发。

但问题在于，很多系统并不是“使用数据库”，而是**被数据库表结构反向支配了设计方式**。

于是系统逐渐变成这样：

```txt
页面表单 -> 接口 -> Service -> ORM -> 数据库表
```

业务行为被压缩成增删改查，系统边界被压缩成表结构，复杂流程被压缩成几个状态字段。久而久之，开发者以为自己在设计系统，实际上只是在给数据库表套一层接口外壳。

所以，“如果没有数据库，系统如何设计？”这个问题的价值，不是让我们真的放弃数据库，而是逼迫我们重新思考：

> 系统的中心到底应该是数据库表，还是业务事实、状态规则和协作过程？

## 一、没有数据库，不等于没有数据

首先要澄清一点：没有数据库，不是说系统不需要存储，也不是说系统不需要一致性、查询和历史记录。

真正要讨论的是：

> 如果不再默认以数据库表作为系统设计的起点，我们会如何组织系统？

传统设计里，很多东西都会被直接塞进表里。

比如订单系统：

```txt
order.status = PAID
order.pay_time = 2026-04-29 12:00:00
```

从表面看，这很清晰。订单状态是已支付，支付时间也记录了。

但如果继续追问，就会发现表字段并没有表达完整的业务过程：

```txt
订单为什么变成已支付？
支付是谁发起的？
支付平台是否回调过多次？
金额是否校验过？
库存是否锁定成功？
支付成功后有没有通知发货？
如果通知失败，系统如何补偿？
```

数据库表通常擅长保存“现在是什么”，但不一定擅长表达“为什么变成现在这样”。

而系统真正复杂的地方，往往不在于保存一个最终状态，而在于处理状态背后的业务过程。

## 二、系统应该先记录事实，而不是先设计表

如果没有数据库，我们首先会被迫思考一个更根本的问题：

> 这个系统里，到底会发生哪些事实？

比如订单系统里，真正重要的可能不是一开始就设计 `orders` 表，而是先识别业务事件：

```txt
OrderCreated
PaymentSubmitted
PaymentSucceeded
PaymentFailed
InventoryLocked
OrderCancelled
OrderShipped
OrderCompleted
```

这些事件表达的是业务世界中已经发生的事实。

相比之下，数据库表更多表达的是某一时刻的结果。

这两者的差别很大。

表记录的是：

```txt
订单现在是已支付
```

事件记录的是：

```txt
订单被创建
用户提交支付
支付平台确认成功
系统校验金额无误
订单状态推进为已支付
```

前者是结果，后者是过程。

一个系统如果只保存结果，短期看很简单；但当业务变复杂后，很多问题都会变得难以解释：

```txt
为什么这个订单状态异常？
为什么库存扣了但订单没发货？
为什么用户付了钱但后台没显示？
为什么这个状态被改过？
是谁改的？
什么时候改的？
依据是什么？
```

所以，好的系统设计不能只关心最终状态，还要关心事实链条。

可以说：

> 表记录的是系统现在相信什么，事件记录的是系统为什么走到现在。

这就是“没有数据库”这个假设带来的第一个启发：系统设计的起点不应该总是表，而应该是业务事实。

## 三、状态不应该只是字段，而应该是规则的结果

很多 CRUD 系统里，状态就是一个字段。

```txt
status = 0
status = 1
status = 2
```

看起来简单，实际上非常危险。

因为字段本身不能说明它为什么可以被修改，也不能说明它是否符合业务规则。

比如订单从“待支付”变成“已支付”，不能只是简单执行：

```sql
UPDATE orders SET status = 'PAID'
```

更合理的思维应该是：

```txt
当 PaymentSucceeded 事件发生时，
系统检查订单是否存在；
检查订单是否仍处于待支付状态；
检查支付金额是否匹配；
检查该支付回调是否已经处理过；
检查订单是否已经超时关闭；
全部满足后，订单状态才能推进为已支付。
```

这时，状态不再是可以随意修改的字段，而是业务规则作用后的结果。

这也是很多系统从“能跑”走向“可靠”的关键。

初级系统关注的是：

```txt
字段怎么改？
```

成熟系统关注的是：

```txt
状态为什么能这样变？
什么情况下不能变？
重复请求怎么办？
失败后如何补偿？
异常状态如何恢复？
```

所以，如果没有数据库，我们反而会更自然地把状态理解成一种**状态机**，而不是一组散落在表里的字段。

## 四、查询不是系统本体，而是面向场景构建的视图

数据库之所以容易成为系统中心，还有一个原因：它同时承担了太多职责。

它既负责持久化，又负责查询；既负责事务，又负责索引；既服务业务接口，又服务后台管理、运营报表、数据分析和审计追踪。

于是很多系统最后变成这样：

```txt
写入查 orders 表
用户页面查 orders 表
后台管理查 orders 表
运营报表也查 orders 表
风控分析还是查 orders 表
```

一开始这样做很高效。后来业务越来越复杂，表也越来越臃肿。

为了适配不同查询场景，表里开始出现越来越多字段：

```txt
展示字段
统计字段
冗余字段
状态字段
临时字段
标记字段
历史兼容字段
```

最后，数据库表不再是清晰的业务模型，而变成了各种需求叠加后的混合物。

如果不以数据库为中心，我们会把查询看成另一件事：

> 写入时，系统关心业务事实是否成立；读取时，系统关心不同场景需要怎样看见这些事实。

也就是说，同一批业务事实，可以生成多个不同视图：

```txt
订单详情视图
用户订单列表视图
商家发货视图
运营统计视图
风控分析视图
财务对账视图
```

这些视图不一定要共享同一种结构。

用户关心的是“我买了什么、什么时候到”；商家关心的是“哪些订单要发货”；财务关心的是“支付、退款、结算是否对得上”；风控关心的是“行为是否异常”。

它们看到的是同一个业务世界，但需要的视角并不相同。

所以，数据库表不应该天然承担所有视图。很多时候，查询模型应该围绕使用场景单独构建，而不是强迫所有场景复用同一张表。

## 五、一致性不只发生在数据库事务里

很多人依赖数据库，是因为数据库事务提供了安全感。

比如：

```txt
开启事务
更新订单
扣减库存
写入支付记录
提交事务
```

在单体系统和简单业务里，这非常有效。

但真实系统复杂起来后，很多动作并不能被一个数据库事务包住。

例如：

```txt
支付平台回调
第三方物流发货
短信通知
异步审核
AI 推理任务
跨服务库存锁定
外部系统同步
```

这些动作涉及外部系统、异步流程、网络失败和重试机制。它们不是一次 `COMMIT` 就能解决的。

这时，系统真正需要的不是单纯依赖数据库事务，而是设计业务协议：

```txt
幂等协议：重复请求不会重复处理
重试协议：失败任务可以安全重试
补偿协议：局部失败后可以回滚或修正
状态机协议：状态只能按合法路径流转
超时协议：长时间未完成的流程自动关闭
对账协议：系统最终能发现并修复不一致
```

数据库事务解决的是局部一致性。

业务协议解决的是系统级一致性。

比如支付回调场景中，数据库事务只能保证“这一次写库不写一半”。但它不能单独回答这些问题：

```txt
支付平台重复回调怎么办？
支付成功但订单已取消怎么办？
金额不一致怎么办？
库存锁定失败怎么办？
订单状态已经变更过怎么办？
下游通知失败怎么办？
```

这些问题必须依靠状态机、幂等键、事件日志、补偿任务和对账机制来解决。

所以，越复杂的系统，越不能只把可靠性寄托在数据库事务上。

## 六、数据库仍然重要，但不应该成为思想中心

讨论“如果没有数据库”，并不是为了否定数据库。

恰恰相反，大多数业务系统仍然应该使用数据库。关系型数据库在事务、一致性、索引、查询、约束和生态成熟度上，仍然是极其可靠的工程选择。

对于很多中小型系统来说，直接使用数据库也是最经济、最稳定、最容易维护的方案。

真正需要警惕的是另一件事：

> 不要因为数据库很好用，就把系统设计退化成表设计。

数据库应该是基础设施，而不是系统的思想中心。

更合理的顺序应该是：

```txt
先理解业务事实
再分析状态流转
再定义业务规则
再设计协作过程
再考虑查询视图
最后决定数据如何落库
```

而不是一上来就问：

```txt
这几个需求要建几张表？
```

表结构当然重要，但它应该是系统建模之后的实现结果，而不应该总是系统设计的起点。

## 七、没有数据库之后，我们才真正开始设计系统

“如果没有数据库，系统如何设计？”这个问题真正有价值的地方在于，它让我们暂时失去一个最熟悉的拐杖。

当不能直接用表承接一切时，我们就必须重新面对系统的本质问题：

```txt
这个业务世界中会发生哪些事实？
哪些事实必须被记录？
状态如何由事实推进？
规则如何被执行？
查询视图如何被构建？
异常如何被恢复？
一致性如何被维护？
历史如何被追溯？
```

这些问题，才是系统设计真正的问题。

数据库可以帮助我们实现系统，但它不能替我们理解系统。

很多 CRUD 系统的问题，不是数据库用错了，而是设计者太早把思考交给了数据库。结果系统看起来有接口、有表、有页面，也能完成基本操作，但缺少对业务事实、状态变化、异常处理和长期演进的表达能力。

好的系统不应该只是数据库表的外壳。

它应该能够清楚表达：

```txt
发生了什么；
为什么发生；
现在处于什么状态；
下一步允许发生什么；
异常情况下如何恢复；
不同角色如何看见同一个业务世界。
```

所以，最终的结论不是“不要数据库”，而是：

> 数据库应该承载数据，但不应该承包思考。
> 系统设计的起点不是表，而是事实、状态、规则和过程。

当我们能在没有数据库的假设下重新设计系统时，回过头再使用数据库，反而会更加清醒。

因为那时数据库不再是系统的中心，而只是系统可靠落地的一种基础设施。
