# 我做了一次多 Agent 架构实验：复杂架构并不天然更聪明

之前我写过几篇关于 AI Agent 架构的文章，核心观点可以概括为一句话：多 Agent 的真正问题，不是给模型分配多少角色，而是如何定义上下文边界、任务边界、协作方式和结果合并机制。

这一次，我做了一个小型实验平台 Agent Architecture Lab，把不同 Agent 架构放在同一批任务下面，观察它们在输出质量、成本、延迟和稳定性上的差异。

实验规模不大，但结果足够有代表性。它让我更确信一个判断：复杂架构不是质量放大器，而是成本、上下文和失败面的放大器。只有当任务本身确实需要拆解、审查或多立场权衡时，多 Agent 架构才可能值得。

## 实验怎么做

这次实验用了四类任务：从零生成简单实验平台方案，基于上一版方案扩展详细设计，对详细设计做审查报告，以及从审查报告中选择一个最有取舍的问题并给出技术判断。

我比较了四种架构：`single`，一个 Agent 直接完成任务；`planner_executor`，Planner 先拆解，Executor 再执行；`planner_executor_reviewer`，Planner 拆解、Executor 写草稿、Reviewer 审查后再修订；`debate`，两个 Debater 给出立场，最后由 Judge 综合。

观察指标包括耗时、token 消耗、估算成本、是否截断、最终答案可用性，以及中间过程是否有复盘价值。它不是统计意义上的大样本实验，但作为第一轮工程观察，已经能暴露典型问题。

## Single Agent 仍然是强基线

很多人谈多 Agent 时，会默认认为 single agent 是低级形态，似乎只要引入 Planner、Reviewer、Debater，系统就会自然变强。

实验结果并不支持这种直觉。`single` 是成本和延迟最低的架构，稳定性也最好，四个任务都没有出现截断。在简单方案任务和明确技术取舍任务中，它的输出质量并不弱。

这说明一个重要工程事实：Single Agent 不是过时方案，而是所有复杂架构必须证明自己值得超过的基线。如果任务边界清楚、输入充分、输出结构明确，增加多个 Agent 很可能只是增加过程成本。

## Planner 有用，但会制造输出膨胀

`planner_executor` 的表现比较微妙。在审查类任务中，它确实有价值，因为审查任务需要覆盖公平性、状态污染、并发限流、成本估算、失败记录、API key 安全、Judge 设计等多个维度。

但在详细设计任务中，Planner 也暴露了副作用：它会推动 Executor 覆盖更多章节，输出更长、更完整，也更容易撞到输出上限。本轮实验中，`planner_executor` 在详细设计任务里出现了 `finish_reason=length`。

Planner 的作用不应该只是让模型多想一点，而应该让任务拆解更可控。如果没有章节预算、长度约束和输出边界，Planner 很容易把任务推向输出膨胀。

## Reviewer 的价值必须被看见

`planner_executor_reviewer` 是这次实验里成本最高、延迟最高的架构。理论上，它适合需要草稿、审查、修订链路的任务，但本轮实验没有体现出与成本匹配的稳定质量跃迁。

尤其是在详细设计任务里，即使引入 Reviewer，最终答案仍然发生截断。Review 本身不能自动解决长文档生成问题，反而会增加中间文本，让后续上下文变长，继续放大 token 消耗和输出压力。

Reviewer 不是没有价值。它的价值在于系统可以记录草稿是什么、审查意见是什么、最终答案改了什么。如果这些信息没有被结构化呈现，用户只能看到最终答案，那么 Reviewer 就只是一次昂贵的额外模型调用。

## Debate 只适合真实取舍

`debate` 架构在技术取舍判断中最有意义。第四个任务要求从审查报告里选择一个最模糊、最有取舍的问题，并给出 A/B 理由、风险、边界、推荐决策和验证方式。

实验中出现的核心问题是 Snapshot Mode vs Chain Mode。一个方向强调公平、隔离和可复现，另一个方向强调连续性、状态传递和真实工作流。让两个 Agent 分别站在不同立场，再由 Judge 综合判断，这种结构是合理的。

但 Debate 不适合所有任务。在简单方案任务中，它容易变成一个 Agent 给方案、另一个 Agent 补风险、Judge 做汇总。看起来像辩论，实际更像方案扩写，增加了成本却未必带来真正的对抗性思考。

## 复杂架构的代价很具体

这次实验里，`planner_executor` 的成本大约是 `single` 的 2.42 倍，`debate` 大约是 3.90 倍，`planner_executor_reviewer` 大约是 6.17 倍。

这还只是模型调用成本。实际工程中，还要考虑结果存储、中间过程展示、失败重试、超时处理、前端可视化、人工复盘成本，以及更复杂的调试路径。

复杂架构一旦引入，就不只是多几个 prompt，而是引入一套新的系统复杂度。多 Agent 不能被当成默认升级路线，它应该是按任务类型选择出来的结构。

## 一套更实际的选择方式

简单方案、边界清晰、输出要求明确的任务，优先使用 `single`。它便宜、快、稳定，而且质量未必差。

中等复杂度、多维度覆盖的任务，可以考虑 `planner_executor`。需要保留草稿、审查、修订证据链的任务，才适合 `planner_executor_reviewer`，但系统必须展示 Review 如何改变最终结果。

存在真实分歧、A/B 取舍、路线判断的问题，适合 `debate`。问题不应该是我要不要使用多 Agent，而应该是任务是否真的需要拆解、是否存在多个立场、是否需要审查证据链、中间过程是否会被使用，以及增加的 token、延迟和失败面是否值得。

## 回到架构本质

多 Agent 架构的价值不是让模型突然变强，而是把复杂任务里的不确定性组织起来。Planner 组织任务结构，Reviewer 组织质量反馈，Debate 组织立场冲突，Judge 组织最终合并。

但这些结构都有成本，也都有失败方式。Planner 可能导致输出膨胀，Reviewer 可能变成昂贵的形式主义，Debate 可能退化成一主一辅的方案拼接，Judge 也可能偏向某一方而不自知。

多 Agent 的成熟，不是从一个 Agent 走向越来越多 Agent，而是知道什么时候应该保持简单，什么时候才值得引入复杂协作。多 Agent 架构不是智能的堆叠，而是复杂性的管理。
