## 项目背景

这是一个以个人技术品牌展示为核心的静态网站项目，目标是把「AI 视觉 + 全栈开发」能力以可读、可演示、可持续更新的形式沉淀下来。  
项目价值不只在页面展示，还在于已经形成了可复用的内容组织模式（首页概览 + 项目详情 + 博客详情 + 编辑器生成流程），可作为后续个人站、技术名片站、轻量内容站的模板资产。

## 当前状态

进行中。

- 已具备能力：  
  - 单页主站（`index.html`）展示完整信息架构（关于、成长历程、技术栈、技能、项目、博客、GitHub、评价、联系）。  
  - 项目列表与详情页体系（`projects/`）可独立承载项目案例。  
  - 博客列表与详情页体系（`blog/`）可独立承载技术文章。  
  - 内容编辑页（`admin/editor.html`）支持博客/项目双模式生成、预览、下载与复制 HTML。

## 技术栈

- 语言：HTML、CSS、JavaScript（原生，无编译步骤）
- 框架：无前端框架（Vanilla 架构）
- 数据存储：无后端数据库；浏览器 `localStorage`（语言偏好）
- 模型 / 外部服务：Cloudflare Worker 表单接口（`js/features.js` 中 `WORKER_URL`）；Google Fonts
- CLI / UI：静态页面 UI + 本地静态服务器（如 `python3 -m http.server`）+ `admin/editor.html` 内容编辑 UI
- 其他依赖：Intersection Observer、requestAnimationFrame、CSS 自定义属性与动画体系

## 核心模块

- 模块1：主站聚合页  
  - 负责什么：聚合个人信息、经历、能力、项目入口、博客入口与联系入口。  
  - 相关文件：`index.html`、`css/base.css`、`css/components.css`、`css/sections.css`、`css/new-sections.css`、`css/effects.css`、`css/responsive.css`

- 模块2：交互动效与页面行为层  
  - 负责什么：Loader、滚动进度、Reveal、计数器、轨道动画、命令面板、表单提交、移动端导航等。  
  - 相关文件：`js/loader.js`、`js/interactions.js`、`js/features.js`、`js/cursor.js`、`js/particles.js`

- 模块3：国际化文案层  
  - 负责什么：中英文文案字典、切换语言、基于 `data-i18n*` 属性注入文本。  
  - 相关文件：`js/i18n.js`、`index.html`（`data-i18n` / `data-i18n-html` 等标记）

- 模块4：项目与博客内容页层  
  - 负责什么：按内容类型组织详情页面，形成可扩展的展示库。  
  - 相关文件：`projects/index.html`、`projects/*.html`、`blog/index.html`、`blog/*.html`、`css/detail.css`、`js/detail.js`

- 模块5：内容编辑与导出层  
  - 负责什么：在浏览器内编辑条目元信息与正文，实时预览并导出 HTML 成品。  
  - 相关文件：`admin/editor.html`

## 关键流程

1. 访问主站 `index.html`，加载基础样式、i18n、loader 与交互脚本。  
2. 用户通过导航与滚动浏览核心区块，交互动效由 `interactions.js` 和 `features.js` 驱动。  
3. 用户从主站进入 `projects/index.html` 或 `blog/index.html`，再跳转到具体详情页。  
4. 详情页复用 `detail.css` + `detail.js`，提供统一的阅读、导航与动效体验。  
5. 内容维护时打开 `admin/editor.html`，选择博客/项目模式并填写内容。  
6. 编辑器生成 HTML 后可直接预览、复制或下载，再人工落盘到 `blog/` 或 `projects/` 目录。

## 关键决策

- 决策：采用纯静态站（无框架、无构建链）  
  - 原因：部署简单、迁移成本低、改动可直接看到结果，适合个人品牌站快速迭代。  
  - 影响：开发门槛低、上线快；但随着页面增长，模块边界与一致性治理压力上升。

- 决策：主站采用「单页聚合 + 分目录详情页」结构  
  - 原因：首页聚焦叙事与转化，详情页承载深度内容，兼顾浏览效率与信息完整度。  
  - 影响：内容组织清晰；但需要维护首页卡片摘要与详情页正文的一致性。

- 决策：自研轻量 i18n（`data-i18n*` + 字典注入）  
  - 原因：无需引入第三方库，适配静态站场景，便于直接在 HTML 上标记。  
  - 影响：实现轻、性能足够；但键名管理和双语文案同步需要人工纪律。

- 决策：为内容生产提供浏览器内编辑器（`admin/editor.html`）  
  - 原因：降低新增文章/项目的操作门槛，减少手写页面骨架重复劳动。  
  - 影响：可复用模板资产沉淀明显；但目前仍是导出后人工落盘，流程尚未自动化。

- 决策：脚本按顺序加载（i18n → loader → 其他交互）  
  - 原因：后续脚本依赖 `currentLang`、`toggleLang()`、`sleep()` 等全局能力。  
  - 影响：运行稳定；但加载顺序变动会引发隐性故障（待通过约束文档或校验脚本固化）。

## 遇到的问题

- 问题：文案数据存在双写点（HTML 默认值 + i18n 字典）  
  - 现象：同一内容在 `index.html` 与 `js/i18n.js` 都有文本，改动容易遗漏。  
  - 影响：中英文本或默认文本可能出现漂移，维护成本上升。  
  - 当前处理：以 i18n 键为主更新，同时同步关键默认文案；尚未自动校验。

- 问题：主站文件体量大、职责集中  
  - 现象：`index.html` 与 `js/i18n.js` 内容规模大，单次修改影响面广。  
  - 影响：回归检查成本高，协同编辑冲突概率增大。  
  - 当前处理：通过 CSS/JS 分层缓解；待确认是否进一步拆分内容数据与模板。

- 问题：视觉样式在多断点下易出现接缝/重叠错觉  
  - 现象：卡片内部模块（如内容区与进度区）在圆角、padding、分隔线组合下可能出现视觉重叠感。  
  - 影响：影响观感与可读性，增加样式调试时间。  
  - 当前处理：通过上/下圆角分离、顶部留白与分隔线优化修正。

## Debug / 踩坑记录

- 现象：交互脚本报错或局部功能失效（语言、终端打字、动效）  
  - 根因：脚本存在全局依赖，若加载顺序被打乱（尤其 `i18n.js`、`loader.js`）会触发未定义引用。  
  - 解决：保持 `index.html` 中脚本顺序稳定，先加载依赖源再加载功能脚本。  
  - 下次如何更快定位：优先检查控制台首个 `ReferenceError`，再核对脚本引入顺序。

- 现象：修改经历/评价文案后，某些语言或默认文案仍显示旧值  
  - 根因：同一文案键在 HTML 默认文本与 i18n 字典中双写，且中英键多。  
  - 解决：按键名全局检索（如 `tl*_`、`test*_`），一次性同步 `index.html` 与 `js/i18n.js`。  
  - 下次如何更快定位：建立“单键改动检查清单”（HTML 默认值、中文字典、英文字典三处）。

- 现象：区块上下拼接处看起来“压叠”  
  - 根因：同色背景 + 双圆角 + 阴影/发光叠加导致视觉边界不清。  
  - 解决：拆分圆角语义（上块仅上圆角、下块仅下圆角），并加入轻量分隔线。  
  - 下次如何更快定位：先看真实 DOM 是否重叠，再看视觉层（圆角、阴影、边框）是否造成错觉。
